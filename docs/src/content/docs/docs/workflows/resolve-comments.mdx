`resolve-comments` fetches a pull request's unresolved review threads, triages
each one, fixes the actionable ones in an isolated worktree, posts a reply on
every thread, and resolves only the threads it actually fixed. It closes the
loop that [pr-review](../pr-review/) opens and that [fix-issue](../fix-issue/)
addresses for fresh issues, without conflating the three jobs.

`pr-review` reads a PR and posts a comment. `fix-issue` implements a GitHub
issue from scratch. `resolve-comments` does neither: it picks up an existing PR
already in review, answers each reviewer thread, and resolves the resolved ones
with evidence.

## Invoke it

```sh
keelson workflow run resolve-comments --inputs ARGUMENTS="resolve comments on 42"
keelson workflow run resolve-comments --inputs ARGUMENTS="address review comments"  # from a PR branch
```

To target an explicit PR from any branch, pass the number. To target the PR
the current branch is linked to, omit the number or use the phrase `CURRENT`.

### What it needs

- `gh` CLI authenticated with **repo write** scope and **GraphQL** access
  (thread `id` and `resolveReviewThread` both require the GraphQL endpoint).
- A GitHub Copilot subscription (`provider: copilot`, `model: auto`).
- A running Keelson server (`keelson start`) for the approval gate.

<Aside type="caution" title="Branch mutation">
This workflow commits, pushes, replies, and resolves. Run it on a PR branch you
own and have push access to. The approval gate pauses before any code lands.
</Aside>

## The shape

The run is eight phases. Two `bash` nodes handle deterministic GitHub I/O
(fetch threads, push, reply/resolve); the judgment nodes are agent `prompt`
steps; one `approval` node is the human gate before any mutation.

| Node | Kind | What it does |
| --- | --- | --- |
| `extract-pr` | prompt | Parses the PR number out of `$ARGUMENTS`, or emits `CURRENT` for the next node to resolve. |
| `fetch-threads` | bash | Resolves the PR number, pages the `reviewThreads` GraphQL query, filters `isResolved == false`, and writes `threads.json` (one entry per unresolved thread with `threadId`, `path`, `line`, `commentId`, and concatenated `body`). |
| `triage` | prompt | Classifies each thread as `actionable-code-change`, `question`, `wontfix`, or `already-addressed`. Pinned to `claude-opus-4.8` for reliable judgment. Writes `triage.json`. |
| `approve` | approval | Shows the triage; pauses for human review before any branch mutation. Captures operator adjustments. |
| `fix` | prompt | Reads triage, applies each `actionable-code-change` fix anchored to `path:line`, runs the narrowest covering check per fix, commits by name (never `git add -A`), and writes `results.json` with `action: "fixed"` or `"replied-only"` per thread. |
| `validate` | bash | Runs `verify.sh` (discovered by the fix step). Always exits 0 so `fix-validation` can read the output. |
| `fix-validation` | prompt | Repairs broken checks if any; commits fixes. Skips if validation passed. |
| `revalidate` | bash | Hard gate: exits non-zero if checks fail or no `verify.sh` was recorded. Prevents push on a broken tree. |
| `push` | bash | `git push` the worktree branch to the existing PR's upstream. |
| `reply-resolve` | prompt | Posts a reply on every thread via REST; calls `resolveReviewThread` via GraphQL only for threads where `results.json` records `action == "fixed"`. |
| `report` | prompt | Summarizes per-thread outcome (resolved / replied / left open) and the final validation status. |

<Aside type="note" title="Honesty invariant">
`reply-resolve` consults `results.json` for every resolution decision. A thread
is resolved if and only if its entry carries `action: "fixed"` — meaning an
actual code commit landed for it. A `question`, `wontfix`, or
`already-addressed` thread gets a reply but stays open. The model cannot
override this: the data drives the mutation, not the reply text.
</Aside>

## Patterns it demonstrates

- **GraphQL for metadata, REST for writes.** GitHub's `reviewThreads` API
  exposes the thread `id` and `isResolved` field only via GraphQL; REST gives
  comment bodies and the `/replies` endpoint. Both layers are used and their
  roles are distinct.
- **Data-driven resolution.** The reply-resolve node reads `results.json`
  rather than re-inferring intent from the thread text. The fix node writes the
  authoritative record; the reply node consumes it. No model judgment in the
  resolve step.
- **Worktree isolation + verify.sh gate.** Mirrors the fix-issue pattern: the
  fix step discovers the project's checks into `verify.sh` (fail-fast,
  `set -euo pipefail`), the validate/revalidate trio gates the push.
- **Commit per thread.** The fix step commits per comment or logical group,
  staging files by name, so the commit history is traceable to the thread that
  motivated each change.
