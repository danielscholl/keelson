Keelson ships seven workflows. They are real, runnable, and the place the
[authoring recipes](../guides/authoring-workflows/) show up composed: a
three-line approval-gate recipe is one thing, the same gate sitting inside a
sixteen-node investigate-and-implement pipeline is where the shape clicks. This
catalog groups what ships by the job it does, and each one gets a full
node-by-node walkthrough.

Each is also readable in a single screen, so opening the file is its own
documentation.

## Where these come from

The seven are **bundled** starters: they ship with the package and seed into
your home (`<home>/workflows/`) on first run, so `keelson workflow list` shows
them as `global` once seeded. Copy and edit one freely; the bundled original
stays as a clean fallback. See [scopes](../reference/workflow-nodes/#scopes) for
how the layers merge.

Run any of them the same way, streaming the DAG as it executes:

```sh
keelson workflow run smoke-test --watch
keelson workflow run fix-issue --inputs ARGUMENTS="42"
```

<Aside type="note" title="What the Copilot workflows need">
Six of the seven set `provider: copilot` and `model: auto`, so they need a
GitHub Copilot subscription registered. The GitHub-facing ones (`fix-issue`,
`pr-review`, `plan-act-evaluate`) also drive the `gh` CLI, so it must be
installed and authenticated. The approval and interactive workflows pause for a
human, which needs a running server (`keelson start`). `smoke-test` is the
exception: it runs provider-free on the stub, with no credentials.
</Aside>

## Ship code

The delivery flagships: an issue taken to a draft PR, a scoped PR review, and a
plan-first lifecycle. Each composes gates, loops, and branches into a real
pipeline, and each has a full walkthrough.

pressure-test a build by pitting a generator against an evaluator.
